# 92. Codex Plus Automation Operation

Status: canonical

## Purpose

Step20 defines the execution surface for running the autonomous loop within a personal ChatGPT Plus plan. The project must not depend on `OPENAI_API_KEY`, `CODEX_API_KEY`, `openai/codex-action`, or other API-metered workers for normal loop operation.

The adopted model is a heartbeat loop:

```text
Codex Cloud / Codex App Automations / Codex CLI
  -> read repository state
  -> run or inspect the controller prompt
  -> implement one Ready ticket or one repair ticket
  -> create a PR

GitHub Actions
  -> run deterministic checks
  -> enforce metadata, path policy, QA, and preflight gates
  -> later auto-merge passing PRs
  -> never call a metered Codex API worker
```

## Non-goal

Step20 does not implement Phase1 gameplay features and does not enable auto-merge. Auto-merge controller work belongs to the next step.

Step20 also does not install a GitHub Actions AI worker. Any workflow that checks out this repository must remain deterministic and must not invoke `openai/codex-action@v1`.

## Plus-plan compatibility rules

| Rule | Required behavior |
|---|---|
| AI execution surface | Codex Cloud, Codex App Automations, or Codex CLI signed in with ChatGPT |
| API key requirement | forbidden for normal loop execution |
| GitHub Actions role | deterministic checks, comments, labels, and later merge/advance only |
| Loop frequency | low-frequency heartbeat, one ticket per run |
| Parallelism | one active implementation ticket by default |
| Usage-limit handling | record wait/block evidence, do not treat it as implementation failure |

## Codex Automation cadence

Recommended initial cadence:

```text
30 to 60 minutes
one ticket per run
no parallel workers
same-ticket repair limit from operations/retry_budget.toml
```

A faster cadence increases usage-limit pressure without improving correctness. The controller therefore treats speed as secondary to reproducibility.

## Automation prompt source

The automation reads one of these prompt surfaces:

1. `reports/loop/<run_id>/next_codex_prompt.md` generated by `taiko_cli loop run-once --mode apply`.
2. `reports/loop/<run_id>/next_codex_prompt.md` generated by `scripts/render_next_codex_prompt.py` when Rust is not available.
3. `prompts/70_codex_automation_loop_runner.md` as the fixed recurring automation prompt.
4. `prompts/71_codex_cloud_ticket_worker.md` as the single-ticket Cloud worker prompt.

The fixed automation prompt is intentionally small. It tells Codex to read project state and then use the generated prompt or controller plan as the specific instruction.

## Human setup required

The human operator does only platform setup:

1. Push this package to a private GitHub repository.
2. Connect the repository to Codex Cloud or Codex App.
3. Configure Codex Cloud setup command as `bash scripts/codex_cloud_setup.sh`.
4. Create a Codex App Automation using `prompts/70_codex_automation_loop_runner.md`.
5. Set cadence to 30 to 60 minutes.
6. Keep agent internet disabled for normal implementation.
7. Do not add `OPENAI_API_KEY` or `CODEX_API_KEY` to this repository for normal operation.

## Expected automation cycle

```text
1. Automation starts.
2. Codex reads AGENTS.md.
3. Codex runs scripts/ci_local_equivalent.sh --static-only when cheap.
4. Codex runs scripts/loop_run_once.sh --mode plan when Rust is available.
5. Codex runs scripts/render_next_codex_prompt.py when Rust is unavailable.
6. Codex follows the generated next_codex_prompt.md.
7. Codex modifies one ticket scope only.
8. Codex opens a PR.
9. GitHub Actions enforce deterministic gates.
10. Later Step21 auto-merge merges only passing PRs.
```

## GitHub review workflow change

`.github/workflows/codex-review.yml` must not call Codex through a GitHub Action in this Plus-plan package. It is retained as a deterministic helper that posts a review request or instructions for a detached Codex Cloud/App review. Actual AI review happens through Codex Cloud/App, not through an API key inside GitHub Actions.

## Usage limit behavior

A Plus usage limit is not a test failure. It is a scheduling condition.

Required response:

```text
verdict = block
category = usage_wait
transition = wait_for_next_codex_window
```

The controller must not create an implementation repair ticket merely because the ChatGPT plan usage window is exhausted.

`OPS-0008 worker handoff` formalizes the handoff surface produced by GitHub Actions after candidate discovery and ticket advance. The deterministic emitter is `scripts/loop_emit_worker_handoff.py`; its primary prompt is `reports/loop/worker_handoff/latest.md` and its machine contract is `reports/loop/worker_handoff/latest.json`.
