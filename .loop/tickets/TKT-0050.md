## 1. Objective

Implement the QA / Regression Gate MVP so a separate QA Session can run fixture validation, headless autoplay, timing analysis, baseline comparison, and verdict normalization as machine-readable JSON.

This ticket exists to prevent Phase1 gameplay feature implementation from relying on manual log inspection or implementation-session self-approval.

## 6. Acceptance criteria

- `qa run` emits `verdict`, `manifest`, `threshold_ms`, component verdicts, required reports, and issues.
- `qa compare` emits `verdict`, `baseline`, `current`, `compared_files`, `missing_current`, `missing_baseline`, and `differing_files`.
- `qa verdict` normalizes an input QA JSON report to `pass`, `reject`, or `block` and emits `next_action`.
- A `reject` can be routed to failure feedback route without human interpretation.
- `GATE-0080` can be evaluated from command outputs and worktree evidence.
- `TKT-0060` remains Blocked until `TKT-0050` is Done and `GATE-0080` has passed; `TKT-0005` remains Blocked until `TKT-0060` is Done and `GATE-0090` has passed.
