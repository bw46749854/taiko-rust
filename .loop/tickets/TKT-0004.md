## 0. Timing analyzer package note

The timing analyzer package includes an initial timing analyzer implementation over headless autoplay evidence so the autonomous loop can produce machine-readable timing metrics before full OpenTaiko-compatible scheduler and audio precision work.

Operational ticket state remains Blocked until `TKT-0003` is formally Done and `GATE-0050` has a pass report. The included implementation is the validation target for `TKT-0004`, not an automatic acceptance of the ticket.

## 1. Objective

Implement the Timing Log Analyzer MVP required before precision tuning and broader Phase1 gameplay tickets proceed. The goal is to make timing-related failures observable by AI sessions through JSON metrics and failure categories, without manual listening, visual inspection, or chart review.

- Add `taiko_timing` timing analyzer MVP.

## 6. Acceptance criteria

- Manifest timing analysis reports `fixture_count = 35`.
- Manifest timing analysis reports `failed_count = 0`.
- Manifest timing analysis reports `analyzed_event_count > 0`.
- Manifest timing analysis reports `max_error_ms <= threshold_ms`.
- Manifest timing analysis reports `mean_error_ms <= threshold_ms`.
- Manifest timing analysis reports `p95_error_ms <= threshold_ms`.
- Passing reports have empty `failure_categories`.
- Single input-file timing analysis verdict is `pass` for valid headless autoplay JSON.
- `timing_log_analyzer` binary alias uses the same JSON contract as `taiko_cli timing analyze`.
- JSON output contains enough fields for QA Session to pass/reject without reading the `.tja` file manually or listening to audio.
