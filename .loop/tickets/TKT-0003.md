## 0. Step9 package note

The Step9 package includes an initial render-free and audio-free perfect autoplay implementation so the autonomous loop can produce machine-readable runtime evidence from committed synthetic fixtures.

Operational ticket state remains Blocked until `TKT-0002` is formally Done and `GATE-0040` has a pass report. The included implementation is the validation target for `TKT-0003`, not an automatic acceptance of the ticket.

## 1. Objective

Implement the Headless Autoplay MVP required before timing-log analyzer, precision tuning, and broader Phase1 gameplay tickets proceed. The goal is to make fixture execution observable by AI sessions without visual inspection, audio playback, or manual chart review.

## 4. Scope

- Add `taiko_runtime` headless perfect autoplay MVP.
- Add `taiko_cli headless autoplay --chart ... --mode perfect --format json`.
- Add `taiko_cli headless autoplay --manifest ... --mode perfect --format json`.
- Make `headless_autoplay` binary a command alias for the same contract.
- Produce deterministic JSON reports with note, hit, miss, fixture, and song-end fields.
- Keep audio, rendering, OpenTaiko-compatible timing precision, score, gauge, branch execution, and golden comparisons out of scope.

## 6. Acceptance criteria

- Manifest headless autoplay reports `fixture_count = 35`.
- Manifest headless autoplay reports `failed_count = 0`.
- Manifest headless autoplay reports `total_note_count > 0`.
- Manifest headless autoplay reports `total_hit_count = total_note_count`.
- Manifest headless autoplay reports `total_miss_count = 0`.
- Every fixture result reports `song_end_reached = true`.
- Single fixture headless autoplay verdict is `pass`.
- `headless_autoplay` binary alias uses the same JSON contract as `taiko_cli headless autoplay`.
- JSON output contains enough fields for QA Session to pass/reject without reading the `.tja` file manually.

## 8. Next-ticket transition evidence

When `TKT-0003` is Done, `GATE-0050` may be evaluated. Passing `GATE-0050` unlocks timing log analyzer MVP work and detailed scheduler/timing tickets that require executable headless runtime evidence.

A reject verdict must produce a failure report in one of these categories:

- `headless_cli_contract_error`
- `headless_fixture_load_error`
- `headless_chart_verdict_error`
- `headless_no_scheduled_notes`
- `headless_autoplay_result_error`

The failure report must route to a repair ticket before timing analyzer, precision tuning, or Phase1 feature implementation proceeds.

## 9. Environment fallback

Rust toolchain unavailable in a validation environment is a validation-environment limitation, not a ticket pass. `scripts/check_headless_autoplay_static.py` only proves package structure and command-surface presence. A Rust-enabled Ticket Implementation or QA Session must still run the cargo checks and the actual `taiko_cli headless autoplay ...` commands before `TKT-0003` can become Done.
