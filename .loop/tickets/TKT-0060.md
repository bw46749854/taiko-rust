## 1. Objective

Create and validate the Phase1 feature loop entry manifest so gameplay feature implementation can begin only after ticket ordering, command evidence, QA verdict requirements, and failure routing are machine-readable.

This ticket exists to prevent `TKT-0005` and downstream gameplay feature tickets from starting based on prose interpretation.

## 6. Acceptance criteria

- Feature manifest validation returns JSON `verdict = pass`.
- Feature plan returns `TKT-0005` as the first gameplay ticket when prerequisites are satisfied, or `block` with explicit missing prerequisites.
- Every manifest ticket has `qa_required = true` and `failure_route_required = true`.
- Every manifest ticket references existing acceptance docs.
- `TKT-0005` remains Blocked until `TKT-0060` is Done and `GATE-0090` has passed.
- A QA `reject` from any listed feature ticket can be routed to Step11 failure feedback without additional human judgement.

## 7. Evidence required

- Implementation plan.
- Plan review result from Design Review Session.
- Command log.
- Feature manifest validation JSON.
- Feature plan JSON.
- QA verdict JSON or block report.
- Worktree separation evidence.
- Final QA / Regression Session verdict.
- Next-ticket transition evidence.

## 8. Next-ticket transition

- `pass`: `GATE-0090` may pass; `TKT-0005` may become eligible according to dependency checks.
- `reject`: create or update a failure report and repair ticket through Step11 commands.
- `block`: keep gameplay feature tickets Blocked until the missing feature-loop evidence exists.
