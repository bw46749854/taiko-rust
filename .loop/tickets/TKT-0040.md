## 0. Step11 package note

The Step11 package includes an initial failure feedback implementation so the autonomous loop can ingest Markdown failure reports, propose repair tickets, and validate repair-ticket shape through deterministic JSON.

Operational ticket state remains Blocked until `TKT-0004` is formally Done and `GATE-0060` has a pass report. The included implementation is the validation target for `TKT-0040`, not an automatic acceptance of the ticket.

## 1. Objective

Implement the Failure Feedback Loop MVP required before Phase1 gameplay feature tickets proceed. The goal is to ensure QA rejects, gate rejects, timing anomalies, parser failures, and CLI contract failures route back into repair tickets without additional human design judgement.

## 6. Acceptance criteria

- Failure ingest JSON reports `report_count >= 1`.
- Failure ingest JSON reports `invalid_count = 0` for the committed sample failure.
- Failure ingest JSON reports `duplicate_keys = []` for the committed sample failure set.
- Parsed failure contains `failure_id`, `category`, `reproduction_command`, `expected_class`, `actual_class`, `proposed_ticket_id`, and `regression_command`.
- Proposed ticket JSON contains `ticket_id`, `source_failure_id`, `repair_scope`, `reproduction_command`, `regression_command`, and `body`.
- Proposed ticket body includes source failure, duplicate key, minimal repair scope, reproduction command, regression command, and acceptance criteria.
- Ticket validation JSON can return `pass` for a repair-ticket-shaped Markdown file and `reject` when required repair fields are missing.
- JSON output contains enough fields for Control Session and QA / Regression Session to route the failure without reading implementation code.

## 9. Environment fallback

Rust toolchain unavailable in a validation environment is a validation-environment limitation, not a ticket pass. `scripts/check_failure_feedback_static.py` only proves package structure and command-surface presence. A Rust-enabled Ticket Implementation or QA Session must still run the cargo checks and actual `taiko_cli loop failure ...` commands before `TKT-0040` can become Done.
