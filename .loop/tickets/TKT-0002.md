## 0. Fixture validation package note

The fixture validation package includes an initial `taiko_cli fixture validate` and `taiko_cli fixture inspect` implementation so the autonomous loop can start validating committed synthetic fixtures through machine-readable JSON.

Operational ticket state remains Blocked until `TKT-0001` is formally Done and `GATE-0030` has a pass report. The included implementation is the validation target for `TKT-0002`, not an automatic acceptance of the ticket.

## 1. Objective

Implement the Fixture Validation MVP required before parser, scheduler, headless autoplay, and timing analyzer tickets proceed. The goal is to make committed synthetic fixture readiness machine-checkable by AI sessions without manual chart inspection.

## 6. Acceptance criteria

- Manifest validation reports `declared_fixture_count = 35`.
- Manifest validation reports `manifest_entry_count = 35`.
- Manifest validation reports `validated_count = 35`.
- Manifest validation reports `missing_count = 0`.
- Manifest validation reports `invalid_count = 0`.
- Manifest validation verdict is `pass`.
- Single fixture inspection verdict is `pass`.
- Unknown TJA commands are reported as errors, not silently ignored.
- JSON output contains enough fields for QA Session to pass/reject without reading the `.tja` file manually.

## 8. Next-ticket transition evidence

When `TKT-0002` is Done, `GATE-0040` may be evaluated. Passing `GATE-0040` unlocks parser expansion and scheduling tickets that depend on synthetic fixture readability.

A reject verdict must produce a failure report in one of these categories:

- `fixture_manifest_error`
- `fixture_file_missing`
- `fixture_tja_structure_error`
- `fixture_unknown_command`
- `fixture_cli_contract_error`

The failure report must route to a repair ticket before `TKT-0003` or any headless autoplay ticket proceeds.

## 9. Environment fallback

Rust toolchain unavailable in a validation environment is a validation-environment limitation, not a ticket pass. `scripts/check_fixture_validation_static.py` only proves package structure and command-surface presence. A Rust-enabled Ticket Implementation or QA Session must still run the cargo checks and the actual `taiko_cli fixture ...` commands before `TKT-0002` can become Done.
